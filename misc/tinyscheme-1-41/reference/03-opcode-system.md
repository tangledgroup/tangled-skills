## opdefines.h Macro Architecture

`opdefines.h` is the single source of truth for all built-in operations. Each line defines one operation via the `_OP_DEF` macro:

```c
#define _OP_DEF(A,B,C,D,E,OP) OP,
#include "opdefines.h"
  OP_MAXDEFINED
#undef _OP_DEF
```

The macro parameters are:
- **A** — Handler function group (`opexe_0` through `opexe_6`)
- **B** — Scheme name (e.g., `"+"`, `"car"`, `"define"`), or `0` for internal opcodes
- **C** — Minimum arity
- **D** — Maximum arity (`INF_ARG = 0xffff` for variadic)
- **E** — Argument type test encoding string
- **OP** — Enum constant name

Example entries:
```c
_OP_DEF(opexe_2, "+",           0,  INF_ARG, TST_NUMBER,      OP_ADD)
_OP_DEF(opexe_2, "car",         1,  1,       TST_PAIR,        OP_CAR)
_OP_DEF(opexe_0, "define",      0,  0,       0,               OP_DEF0)
_OP_DEF(opexe_5, "read",        0,  1,       TST_INPORT,      OP_READ)
```

The file is included three times with different `_OP_DEF` definitions to generate:
1. The `enum scheme_opcodes` in `scheme-private.h`
2. The `dispatch_table[]` array in `scheme.c`
3. (Implicitly) the handler function signatures

## Enum Generation

In `scheme-private.h`, the enum is generated by redefining `_OP_DEF` to emit enum values:

```c
enum scheme_opcodes {
#define _OP_DEF(A,B,C,D,E,OP) OP,
#include "opdefines.h"
  OP_MAXDEFINED
};
```

This produces an enum where each opcode has a unique integer value matching its index in the dispatch table.

## Dispatch Table Structure

In `scheme.c`, the dispatch table maps opcodes to handler functions and metadata:

```c
typedef struct {
  dispatch_func func;          /* opexe_N function pointer */
  char *name;                  /* Scheme name or NULL for internal */
  int min_arity;               /* minimum argument count */
  int max_arity;               /* maximum argument count (INF_ARG = variadic) */
  char *arg_tests_encoding;    /* type test string or NULL */
} op_code_info;

static op_code_info dispatch_table[] = {
#define _OP_DEF(A,B,C,D,E,OP) {A,B,C,D,E},
#include "opdefines.h"
  { 0 }
};
```

Built-in procedures (those with a non-NULL name) are registered in the global environment during `scheme_init()`:

```c
for(i = 0; i < n; i++) {
    if(dispatch_table[i].name != 0) {
      assign_proc(sc, (enum scheme_opcodes)i, dispatch_table[i].name);
    }
}
```

Each built-in procedure cell stores its opcode index as an integer in the number field:
```c
static pointer mk_proc(scheme *sc, enum scheme_opcodes op) {
     pointer y = get_cell(sc, sc->NIL, sc->NIL);
     typeflag(y) = (T_PROC | T_ATOM);
     ivalue_unchecked(y) = (long)op;
     set_num_integer(y);
     return y;
}
```

## Test Predicates and Type Checking

The `arg_tests_encoding` field is a string of characters, each encoding a type predicate for one argument position. The last test is replicated for all remaining arguments (supporting variadic functions).

```c
static struct {
  test_predicate fct;
  const char *kind;
} tests[] = {
  {0, 0},              /* TST_NONE = '\0' */
  {is_any, 0},         /* TST_ANY    = '\1' */
  {is_string, "string"},     /* TST_STRING = '\2' */
  {is_symbol, "symbol"},     /* TST_SYMBOL = '\3' */
  {is_port, "port"},
  {is_inport, "input port"},
  {is_outport, "output port"},
  {is_environment, "environment"},
  {is_pair, "pair"},
  {0, "pair or '()"},        /* TST_LIST = '\11' */
  {is_character, "character"},
  {is_vector, "vector"},
  {is_number, "number"},
  {is_integer, "integer"},
  {is_nonneg, "non-negative integer"},
};
```

For `OP_ADD` (`"+"`), the encoding is `TST_NUMBER` — all arguments must be numbers. The test loop in `Eval_Cycle()` iterates through args, applying each test character, then repeating the last test for any remaining variadic args.

## Eval_Cycle: The Main Loop

```c
static void Eval_Cycle(scheme *sc, enum scheme_opcodes op) {
  sc->op = op;
  for (;;) {
    op_code_info *pcd = dispatch_table + sc->op;

    /* If built-in function (has a name), validate arguments */
    if (pcd->name != 0) {
      int n = list_length(sc, sc->args);
      /* Check arity bounds */
      if(n < pcd->min_arity || n > pcd->max_arity) { error; }
      /* Run type tests on each argument */
      /* ... iterates through arg_tests_encoding ... */
    }

    ok_to_freely_gc(sc);  /* Clear recent alloc chain */

    if (pcd->func(sc, (enum scheme_opcodes)sc->op) == sc->NIL) {
      return;  /* Error or quit — exit the loop */
    }

    if(sc->no_memory) {
      fprintf(stderr, "No memory!\n");
      return;
    }
    /* Loop continues with sc->op set by s_return() or s_goto() */
  }
}
```

The loop runs indefinitely. Each iteration:
1. Looks up the current opcode in the dispatch table
2. Validates arguments (for named built-ins only)
3. Calls the handler function
4. Handler either returns `sc->T` (continue loop with new `sc->op`) or `sc->NIL` (error/quit, exit)

The `s_goto(op)` macro sets `sc->op` and returns `sc->T`, causing the next iteration to dispatch to the new opcode. The `_s_return(value)` function pops the dump stack, restores registers, and sets `sc->op` for the caller's continuation.

## Handler Functions (opexe_0 through opexe_6)

Seven handler functions organize opcodes by category:

- **`opexe_0`**: Control flow — LOAD, T0LVL/T1LVL (REPL), EVAL, E0ARGS/E1ARGS (argument evaluation), APPLY, LAMBDA, QUOTE, DEF0/DEF1 (define), SET0/SET1 (set!), BEGIN, IF0/IF1, LET0/LET1/LET2, LET* variants, GENSYM, VALUEPRINT
- **`opexe_1`**: Special forms — LETREC, COND0/COND1, DELAY, AND0/AND1, OR0/OR1, cons-stream, MACRO0/MACRO1, CASE0/CASE1/CASE2, apply, eval, call/cc
- **`opexe_2`**: Arithmetic and data operations — +, -, *, /, quotient, remainder, modulo, car, cdr, cons, set-car!, set-cdr!, string ops, vector ops, char ops, type conversions
- **`opexe_3`**: Predicates — not, boolean?, null?, =, <, >, <=, >=, symbol?, number?, string?, integer?, real?, char?, port?, input-port?, output-port?, procedure?, pair?, list?, eq?, eqv?, char classifiers
- **`opexe_4`**: I/O and utilities — force, write, display, newline, error, reverse, append, put/get (plists), quit, gc, open/close ports, string ports, environments
- **`opexe_5`**: Reader and writer — READ, read-char, peek-char, set-input/output-port, RDSEXPR/RDLIST/RDDOT (S-expression reading), RDQUOTE/RDQQUOTE/RDUNQUOTE (quasiquote), P0LIST/P1LIST/PVECFROM (printing)
- **`opexe_6`**: Utilities — length, assq, get-closure-code, closure?, macro?

## Syntax vs Procedure Dispatch

Syntax objects (special forms) are distinguished from procedures by the `T_SYNTAX` flag. During evaluation of a pair in `OP_EVAL`:

```c
if (is_syntax(x = car(sc->code))) {     /* SYNTAX */
    sc->code = cdr(sc->code);
    s_goto(sc, syntaxnum(x));
} else {                                  /* PROCEDURE */
    s_save(sc, OP_E0ARGS, sc->NIL, sc->code);
    sc->code = car(sc->code);
    s_goto(sc, OP_EVAL);
}
```

`syntaxnum()` maps the syntax symbol to its opcode using a hard-coded switch on string length and characters:

```c
static int syntaxnum(pointer p) {
     const char *s = strvalue(car(p));
     switch(strlength(car(p))) {
     case 2: if(s[0]=='i') return OP_IF0; else return OP_OR0;
     case 3: if(s[0]=='a') return OP_AND0; else return OP_LET0;
     case 4: switch(s[3]) { case 'e': return OP_CASE0; ... }
     /* ... more cases for begin, quote, lambda, define, letrec, cons-stream */
     }
}
```

This hard-coded mapping is efficient but requires updating if new syntax forms are added. Syntax forms bypass argument evaluation and type checking — they receive raw code to interpret according to their semantics.
