# Internals and Architecture

## Bytecode

The compiler generates stack-based bytecode directly with no intermediate representation such as a parse tree. This makes compilation very fast. Several optimization passes are performed over the generated bytecode.

Key design decisions:

- **Stack-based** — Simple and generates compact code
- **Compile-time stack sizing** — Maximum stack size per function computed at compile time, so no runtime stack overflow tests needed
- **Compressed line number table** — Separate from bytecode for debug information
- **Closure optimization** — Access to closure variables is almost as fast as local variables
- **Direct eval in strict mode** — Optimized path

## Executable Generation with qjsc

The `qjsc` compiler generates C source from JavaScript files. By default the C source is compiled with the system compiler (`gcc` or `clang`).

### Compilation Modes

```bash
# Full standalone executable
./qjsc -o hello examples/hello.js
./hello

# Bytecode in a C file (no main)
./qjsc -c -o out.c examples/hello.js

# C file with main() and bytecode
./qjsc -e -o out.c examples/hello.js

# Compile as module
./qjsc -m -o app module.mjs

# Include external C module
./qjsc -M my_module.so -o app main.js
```

### Size Optimization

Disable specific JavaScript features to produce smaller executables:

```bash
./qjsc -fno-eval -fno-regexp -fno-proxy -fno-bigint -o tiny script.js
```

Available `-fno-*` options: `eval`, `string-normalize`, `regexp`, `json`, `proxy`, `map`, `typedarray`, `promise`, `bigint`.

Link Time Optimization (`-flto`) produces smaller and faster executables but compiles slower. Automatically enabled when `-fno-*` options are used.

### Dynamic Module Declaration

When code uses `import()` or `os.Worker()`, the compiler cannot statically find module names. Use `-D` to declare them:

```bash
./qjsc -D dynamic_module.mjs -o app main.js
```

### Binary JSON

`qjsc` serializes compiled scripts/modules to a binary format. A subset of this format (without functions or modules) can be used as binary JSON. The `test_bjson.js` example demonstrates usage.

Warning: the binary JSON format may change without notice — do not use for persistent storage.

## Strings

Stored as either 8-bit or 16-bit character arrays, enabling fast random access. The C API provides functions to convert JavaScript strings to C UTF-8 encoded strings. When the string contains only ASCII characters, no copying is involved.

## Objects

- **Shared shapes** — Object prototype, property names, and flags are shared between objects to save memory
- **Dense arrays** — Arrays with no holes (except at end) are optimized
- **TypedArray optimization** — Direct access to typed array data

## Atoms

Object property names and some strings are stored as Atoms (unique strings) for fast comparison. Represented as 32-bit integers. Half the atom range is reserved for immediate integer literals from 0 to 2^31-1.

```c
JSAtom atom = JS_NewAtom(ctx, "propertyName");
JSValue val = JS_AtomToValue(ctx, atom);
JS_FreeAtom(ctx, atom);
```

## Numbers

Represented as either 32-bit signed integers or 64-bit IEEE-754 floating point. Most operations have fast paths for the 32-bit integer case.

## Garbage Collection

Reference counting provides automatic and deterministic object freeing. A separate cycle removal pass runs when allocated memory grows too large. The cycle removal algorithm uses only reference counts and object content — no explicit GC roots need to be managed in C code.

Call `JS_RunGC()` to force collection, or use `std.gc()` from JavaScript.

## JSValue Representation

**64-bit mode:** JSValue is 128-bit (two CPU registers). No NaN boxing used — memory usage is less critical on 64-bit systems.

**32-bit mode:** NaN boxing stores 64-bit floating point numbers within the same register space. Optimized so that 32-bit integers and reference-counted values can be efficiently tested.

In both modes, JSValue fits exactly two CPU registers for efficient return from C functions.

Value tags:

- `JS_TAG_INT` (0) — 32-bit integer
- `JS_TAG_BOOL` (1) — Boolean
- `JS_TAG_NULL` (2) — null
- `JS_TAG_UNDEFINED` (3) — undefined
- `JS_TAG_FLOAT64` (8) — Double precision float
- `JS_TAG_BIG_INT` (-9) — BigInt
- `JS_TAG_SYMBOL` (-8) — Symbol
- `JS_TAG_STRING` (-7) — String
- `JS_TAG_OBJECT` (-1) — Object
- `JS_TAG_EXCEPTION` (6) — Exception marker

All negative tags have reference counts.

## Function Calls

The engine is optimized for fast function calls. The system stack holds JavaScript parameters and local variables directly — no separate VM stack allocation needed.

## RegExp Engine

A custom regular expression engine, both small and efficient, supporting all ES2023 features including Unicode properties. Like the JavaScript compiler, it generates bytecode directly without a parse tree.

Backtracking uses an explicit stack (no system stack recursion). Simple quantifiers are specifically optimized to avoid recursion. The full regexp library weighs about 15 KiB of x86 code (excluding Unicode library).

As of 2025-09-13: supports the `v` flag (Unicode sets), RegExp modifiers, and `RegExp.escape()`.

## Unicode Library

A custom Unicode library eliminates dependency on external libraries like ICU. All Unicode tables are compressed while maintaining reasonable access speed. Supports:

- Case conversion
- Unicode normalization
- Unicode script queries
- Unicode general category queries
- All Unicode binary properties

Updated to Unicode 15.0.0 as of 2023-12-09 release. The full library weighs about 45 KiB of x86 code.

## BigInt

BigInts use binary two's complement notation. An additional short BigInt value type optimizes performance for small BigInt values (32 or 64 bits depending on platform).

As of 2025-04-26, a new BigInt implementation optimized for small numbers replaced the previous bignum extensions. The `qjscalc` application was removed — use the [BFCalc calculator](https://bellard.org/libbf) instead.

## Sub-projects

QuickJS embeds C libraries usable independently:

- **libregexp** — Small, fast regexp library fully compliant with ES2023
- **libunicode** — Unicode library supporting case conversion, normalization, script/category queries, and binary properties
- **dtoa** — Float64 printing and parsing library

## Installation

```bash
# Clone from GitHub
git clone https://github.com/bellard/quickjs.git
cd quickjs

# Build
make

# Install to /usr/local (optional)
sudo make install

# Run built-in tests
make test

# Run test262 suite
git clone https://github.com/tc39/test262.git test262
cd test262 && patch -p1 < ../tests/test262.patch && cd ..
make test2
```

Edit the top of the `Makefile` to select specific build options. On some OSes, add `-latomics` to the `LIBS` variable or disable `CONFIG_ATOMICS` in `quickjs.c`.

Cross-compilation support available for Windows via MinGW and Cosmopolitan builds for multi-platform binaries (Linux, Mac, Windows, FreeBSD, OpenBSD, NetBSD on ARM64 and x86_64).

## Command Line Tools

### qjs (Interpreter)

```bash
./qjs examples/hello.js          # Run a script
./qjs -e "print(1+2)"            # Evaluate expression
./qjs -i                         # Interactive REPL
./qjs -m module.mjs              # Load as ES6 module
./qjs --std script.js            # Make std/os available in non-module
./qjs -d script.js               # Dump memory usage stats
./qjs -I helper.js main.js       # Include additional file
./qjs --strip-source script.js   # Strip source from output
```

Module autodetection: `.mjs` extension or first keyword being `import`.

### qjsc (Compiler)

```bash
./qjsc -o hello examples/hello.js        # Standalone executable
./qjsc -c -o out.c examples/hello.js     # C file with bytecode only
./qjsc -e -o out.c examples/hello.js     # C file with main() + bytecode
./qjsc -m -o app module.mjs              # Compile as module
./qjsc -N mydata -o out.c script.js      # Set C data name
./qjsc -M ext.so -o app main.js          # Add external C module
./qjsc -flto -fno-eval -o tiny script.js # LTO + feature stripping
```
