# Scaling Patterns for Long-Running Agents

How to run coding agents autonomously for weeks at a time, writing millions of lines of code across hundreds of concurrent agents.

## The Challenge

Single agents work well for focused tasks but are slow for complex projects. Scaling means:

- **Time**: Running for weeks without human intervention
- **Parallelism**: Hundreds of agents working concurrently
- **Code volume**: Millions of lines across thousands of files
- **Token usage**: Trillions of tokens over the lifetime

**Cursor's experiments** pushed these boundaries to understand limits and discover patterns.

## Architecture Patterns

### 1. Planner-Worker Hierarchy

The most successful pattern separates planning from execution:

```
┌─────────────────────────────────────────┐
│              Planners                   │
│  - Explore codebase continuously        │
│  - Create tasks from high-level goals   │
│  - Spawn sub-planners recursively       │
│  - Parallel and recursive planning      │
└──────────────┬──────────────────────────┘
               │ Creates tasks
               ▼
┌─────────────────────────────────────────┐
│            Task Queue                   │
│  - Unassigned tasks with clear          │
│    definitions and acceptance criteria  │
└──────────────┬──────────────────────────┘
               │ Workers pick up
               ▼
┌─────────────────────────────────────────┐
│              Workers                    │
│  - Pick tasks and focus entirely        │
│  - No coordination with other workers   │
│  - Grind until complete, then push      │
└──────────────┬──────────────────────────┘
               │ Cycle complete
               ▼
┌─────────────────────────────────────────┐
│               Judge                      │
│  - Evaluate progress at cycle end       │
│  - Determine whether to continue        │
│  - Start next iteration fresh           │
└─────────────────────────────────────────┘
```

**Why it works**:
- Separation of concerns: planners think, workers do
- No coordination overhead between workers
- Parallel planning through sub-planners
- Fresh starts combat drift

### 2. Lock-Free Coordination

Instead of explicit locks, use optimistic concurrency:

```python
class OptimisticTaskQueue:
    def claim_task(self, agent_id: str) -> Optional[Task]:
        tasks = self.read_tasks()
        
        for task in tasks:
            if task['status'] == 'available':
                # Optimistic assignment
                new_version = task['version'] + 1
                if self.write_task_optimistic(
                    task['id'], 
                    {'status': 'in_progress', 'assigned_to': agent_id, 'version': new_version},
                    expected_version=task['version']
                ):
                    return task
        
        return None
    
    def write_task_optimistic(self, task_id: str, updates: dict, expected_version: int) -> bool:
        current = self.read_task(task_id)
        if current['version'] != expected_version:
            return False  # Conflict, retry
        
        current.update(updates)
        self.write_task(task_id, current)
        return True
```

**Benefits over locks**:
- No deadlocks or lock contention
- Agents don't wait, they retry
- Simpler failure recovery (no stuck locks)

**Trade-offs**:
- More retries under high contention
- Need idempotent task claiming

### 3. Git-Based Synchronization

Use git as coordination mechanism:

```bash
#!/bin/bash
# Each agent runs in Docker container with shared repo

while true; do
    # Pull latest changes
    git pull origin main
    
    # Pick unassigned task
    TASK=$(ls current_tasks/*.pending | head -1)
    if [ -z "$TASK" ]; then
        sleep 30
        continue
    fi
    
    # Claim task by creating lock
    mv "$TASK" "${TASK%.pending}.locked-$$"
    
    # Create feature branch
    TASK_NAME=$(basename "$TASK" .locked-$$)
    git checkout -b "task/$TASK_NAME"
    
    # Do the work (agent loop)
    claude --prompt "$(cat agent-prompt.md)"
    
    # Commit and push
    git add -A
    git commit -m "Complete $TASK_NAME"
    git push origin "task/$TASK_NAME"
    
    # Create PR or merge
    gh pr create --title "$TASK_NAME" --body "Auto-generated by agent"
    
    # Release lock
    rm "${TASK%.pending}.locked-$$"
done
```

**Why git works**:
- Natural version control and rollback
- Branches isolate work
- PRs provide review boundary
- Distributed, no central coordinator

### 4. Periodic Fresh Starts

Combat drift and tunnel vision with periodic resets:

```python
async def run_with_fresh_starts(max_runtime_hours: int = 24):
    start_time = time.time()
    
    while not await goal_achieved():
        # Run agents for max_runtime_hours
        await run_agents(timeout=max_runtime_hours * 3600)
        
        # Check if time limit reached
        if time.time() - start_time > max_runtime_hours * 3600:
            # Summarize progress so far
            summary = await summarize_progress()
            
            # Log metrics before reset
            await log_metrics()
            
            # Fresh start with summarized context
            start_time = time.time()
            await spawn_fresh_agents(summary)
        
        # Also check for goal achievement
        if await judge.evaluate():
            break
```

**What to summarize**:
- What was accomplished
- Current state of codebase
- Known issues and blockers
- Remaining work

**What to discard**:
- Conversation history (causes drift)
- Failed attempts (learned lessons in summary)
- Temporary context (rebuild from codebase)

## Model Selection Strategies

### Role-Based Model Assignment

Different models excel at different roles:

```python
def select_model_for_role(role: str, task_complexity: str) -> str:
    model_matrix = {
        ('planner', 'high'): 'gpt-5.2',      # Best at planning
        ('planner', 'low'): 'gpt-5.1-codex',  # Good enough for simple planning
        
        ('worker', 'high'): 'gpt-5.1-codex',  # Trained for coding
        ('worker', 'low'): 'claude-3-haiku',  # Cheap for simple tasks
        
        ('reviewer', 'high'): 'claude-opus',   # Best at code review
        ('reviewer', 'low'): 'gpt-4',          # Good enough
        
        ('judge', 'any'): 'gpt-5.2',           # Needs to maintain focus
    }
    
    return model_matrix.get((role, task_complexity), 'gpt-5.1-codex')
```

### Model Characteristics

**GPT-5.2 (Extended Thinking)**:
- Best for: Long-running autonomous work
- Strengths: Following instructions, keeping focus, avoiding drift, precise implementation
- Use for: Planners, judges, complex tasks

**Opus 4.5**:
- Behavior: Tends to stop earlier, take shortcuts, yield control quickly
- Best for: Tasks requiring human collaboration
- Avoid for: Fully autonomous long-running work

**GPT-5.1-Codex**:
- Trained specifically for coding
- Good for: Worker agents implementing tasks
- Not best for: Planning (surprisingly)

**Claude Haiku**:
- Cheap and fast
- Good for: Simple worker tasks, summarization
- Avoid for: Complex reasoning

## Test-Driven Development

### High-Quality Test Harnesses

Tests define what the agent solves. Critical patterns:

```python
# test_runner.py

async def run_tests(fast_mode: bool = False):
    if fast_mode:
        # Run deterministic subsample
        return await run_deterministic_subset()
    else:
        # Run full suite
        return await run_full_suite()

async def run_deterministic_subset():
    # Select tests that:
    # 1. Have no randomness
    # 2. Run quickly (< 1 second each)
    # 3. Cover critical paths
    
    tests = select_deterministic_tests()
    results = await run_parallel(tests, max_workers=10)
    
    # Print minimal output (context window rotting is real)
    print(f"Passed: {results.passed}/{results.total}")
    
    # Log details to file
    with open('test_details.log', 'w') as f:
        for test in results.failed:
            f.write(f"FAILED: {test.name}\n{test.output}\n")
    
    return results

def select_deterministic_tests():
    # Exclude tests with:
    # - Random number generation
    # - Network calls
    # - File system dependencies
    # - Long runtimes
    
    all_tests = discover_tests()
    return [t for t in all_tests if is_deterministic(t) and runtime(t) < 1.0]
```

### Test Output Patterns

**Bad**: Verbose output that fills context window

```
Test 1: test_addition ... ok (0.023s)
Test 2: test_subtraction ... ok (0.019s)
Test 3: test_multiplication ... FAILED (0.045s)
  File "test_math.py", line 42, in test_multiplication

  AssertionError: Expected 6, got 7
  Stack trace:
    at multiply (math.js:15)
    at test_multiplication (test_math.py:42)
    ...
```

**Good**: Minimal output, details in files

```

```

## Code Quality Management

### The Degradation Problem

Anthropic's C compiler case study found: **Code quality degrades as complexity grows**. New features frequently broke existing functionality even with CI.

### Mitigation Strategies

#### 1. Agent Specialization

Dedicate agents to specific roles:

#### 2. Oracle-Based Validation

Use known-good implementations as oracles:

```python
async def validate_with_oracle(agent_code: str, oracle_code: str):
    # Run same tests against both
    agent_results = await run_tests(agent_code)
    oracle_results = await run_tests(oracle_code)
    
    # Compare outputs
    for test_name in agent_results.keys():
        if agent_results[test_name] != oracle_results[test_name]:
            return {
                'passed': False,
                'mismatch': test_name,
                'agent_output': agent_results[test_name],
                'oracle_output': oracle_results[test_name],
            }
    
    return {'passed': True}

# Example: Use GCC as oracle for C compiler
async def validate_c_compiler(agent_compiler: str):
    test_programs = load_test_programs()
    
    for program in test_programs:
        # Compile with both
        agent_binary = await compile(agent_compiler, program)
        gcc_binary = await compile('gcc', program)
        
        # Run both
        agent_output = await run(agent_binary)
        gcc_output = await run(gcc_binary)
        
        if agent_output != gcc_output:
            return f"Mismatch on {program.name}: expected {gcc_output}, got {agent_output}"
    
    return "All tests passed"
```

Start simple, add complexity gradually:

## Cost Management

### Anthropic C Compiler Case Study

- **Total cost**: Just under $20k
- **Code produced**: Roughly 100k lines
- **Cost per line**: ~$0.20
- **Comparison**: Cost-effective vs human teams, but passing tests doesn't guarantee quality without human verification

### Cost Optimization Strategies

#### 1. Model Routing

```python
def select_model_by_task_complexity(task: str) -> str:
    complexity = estimate_complexity(task)
    
    if complexity < 0.3:
        return 'claude-3-haiku'  # $0.25/M tokens
    elif complexity < 0.7:
        return 'gpt-4'           # $5/M tokens
    else:
        return 'gpt-5.2'         # $20/M tokens but worth it for complex tasks
```

#### 2. Prompt Caching

Cache frequently-used context:

```python
async def create_message_with_cache(user_prompt: str):
    # Cache system prompt and documentation
    return await anthropo.messages.create(
        model='claude-3-5-sonnet',
        messages=[
            {
                'role': 'user',
                'content': [
                    {
                        'type': 'document',
                        'source': {
                            'type': 'text',
                            'media_type': 'text/markdown',
                            'data': load_documentation(),
                        },
                        'cache_control': {'type': 'ephemeral'},  # Cache this
                    },
                    {'type': 'text', 'text': user_prompt},
                ]
            }
        ],
    )
```

**Savings**: Cached content doesn't count against context window and repeated access is cheaper.

#### 3. Parallelism vs Serial

More agents in parallel can be cheaper than one agent running longer:

```
Single agent: 100 hours × $10/hour = $1,000
10 agents: 10 hours × 10 agents × $10/hour = $1,000 (but done 10x faster)
```

**Key insight**: Parallelism reduces wall-clock time without increasing cost (if agents are utilization-efficient).

## Monitoring and Observability

### Metrics to Track

```python
class AgentMetrics:
    def __init__(self):
        self.tasks_completed = 0
        self.tasks_failed = 0
        self.lines_written = 0
        self.tokens_used = 0
        self.cost_incurred = 0.0
        self.avg_task_duration = 0.0
    
    async def log_task_complete(self, task: Task, duration: float, tokens: int, cost: float):
        self.tasks_completed += 1
        self.lines_written += task.lines_added
        self.tokens_used += tokens
        self.cost_incurred += cost
        
        # Update moving average
        self.avg_task_duration = (self.avg_task_duration * self.tasks_completed + 
                                   duration) / (self.tasks_completed + 1)
    
    def get_efficiency(self) -> float:
        return self.lines_written / self.cost_incurred if self.cost_incurred > 0 else 0
    
    def get_success_rate(self) -> float:
        total = self.tasks_completed + self.tasks_failed
        return self.tasks_completed / total if total > 0 else 0
```

### Alerting Conditions

```python
def check_health(metrics: AgentMetrics) -> list[str]:
    alerts = []
    
    if metrics.success_rate < 0.7:
        alerts.append(f"Low success rate: {metrics.success_rate:.2%}")
    
    if metrics.get_efficiency() < 10:  # Less than 10 lines per dollar
        alerts.append(f"Low efficiency: {metrics.get_efficiency():.2f} lines/$")
    
    if metrics.avg_task_duration > 3600:  # Tasks taking > 1 hour
        alerts.append(f"Long task duration: {metrics.avg_task_duration/60:.1f} minutes")
    
    if metrics.cost_incurred > BUDGET_LIMIT * 0.8:
        alerts.append(f"Approaching budget limit: ${metrics.cost_incurred:.2f}")
    
    return alerts
```

## Key Lessons

### 1. Simplicity Wins

Many improvements came from **removing complexity** rather than adding it:

- Initially built integrator role for quality control
- Found it created more bottlenecks than it solved
- Workers were already capable of handling conflicts themselves

**Lesson**: The best system is often simpler than you'd expect.

### 2. Right Amount of Structure

- Too little structure → agents conflict, duplicate work, drift
- Too much structure → fragility and bottlenecks
- Right amount is somewhere in the middle

### 3. Prompts Matter Most

Surprising amount of system's behavior comes down to how agents are prompted. Getting them to:
- Coordinate well
- Avoid pathological behaviors  
- Maintain focus over long periods

Required extensive experimentation. **The harness and models matter, but prompts matter more.**

### 4. Tests Define What Agents Solve

High-quality test harnesses are critical. Tests should:
- Print minimal output (context window rotting)
- Log details to files
- Have fast modes for quick feedback
- Be deterministic when possible

### 5. Humans Still Needed

Passing tests doesn't guarantee quality without human verification. Code efficiency was poor (worse than `gcc -O0`). New features broke existing functionality even with CI.

**Bottom line**: Agents can write code at scale, but humans need to verify quality and steer direction.

3. **Git-based synchronization** - Natural version control and isolation
4. **Periodic fresh starts** - Combat drift with summarized context resets

6. **Test-driven development** - Tests define what agents solve
7. **Agent specialization** - Dedicated agents for consolidation, optimization, documentation
8. **Oracle-based validation** - Use known-good implementations as reference
9. **Cost optimization** - Model routing, prompt caching, efficient parallelism
10. **Humans steer, agents execute** - Scale is possible but verification still needed
