# Session Lifecycle

## Overview

Sessions represent a specific conversation or thread between client and agent. Each session maintains its own context, conversation history, and state. Multiple independent sessions can run with the same agent.

Before creating sessions, clients must complete the initialization phase.

## Creating a Session

Clients create a new session via `session/new`:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "session/new",
  "params": {
    "cwd": "/home/user/project",
    "mcpServers": [
      {
        "name": "filesystem",
        "command": "/path/to/mcp-server",
        "args": ["--stdio"],
        "env": []
      }
    ]
  }
}
```

Parameters:
- **cwd** — Working directory for the session (required)
- **mcpServers** — MCP servers the agent should connect to (optional, empty array if none)

Response includes a unique session ID:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "sessionId": "sess_abc123def456"
  }
}
```

The agent connects to MCP servers and creates session context upon receiving this request.

## Loading Sessions

Agents supporting the `loadSession` capability allow clients to resume previous conversations. This enables persistence across restarts and sharing sessions between client instances.

Check support by verifying `loadSession: true` in the initialize response, then call:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "session/load",
  "params": {
    "sessionId": "sess_789xyz",
    "cwd": "/home/user/project",
    "mcpServers": []
  }
}
```

During load, the agent restores session context, reconnects to MCP servers, and replays conversation history via `session/update` notifications before returning the response.

## Resuming Sessions

The `session/resume` method restores a session without replaying full conversation history. This is faster than `session/load` but provides less context:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "session/resume",
  "params": {
    "sessionId": "sess_789xyz"
  }
}
```

## Listing Sessions

Agents supporting `sessionCapabilities.list` allow clients to discover existing sessions:

```json
{
  "jsonrpc": "2.0",
  "id": 4,
  "method": "session/list",
  "params": {
    "cwd": "/home/user/project",
    "cursor": "eyJwYWdlIjogMn0="
  }
}
```

Support optional filtering by working directory and pagination via cursor.

## Session Info Updates

Agents can push session metadata updates in real-time via the `session_info_update` notification, keeping session titles and metadata synchronized without polling.

## Closing Sessions

Sessions are closed when:
- The agent process terminates
- The client closes the connection
- Explicitly via `session/close` (if supported)

## Session ID Format

Session IDs are opaque strings generated by the agent. Clients should treat them as identifiers and not make assumptions about their format.
