# Agent2Agent (A2A) Protocol 1.0.0

## Overview

The Agent2Agent (A2A) Protocol is an open standard designed to facilitate communication and interoperability between independent, potentially opaque AI agent systems. In an ecosystem where agents might be built using different frameworks, languages, or by different vendors, A2A provides a common language and interaction model.

A2A enables agents to:

- Discover each other's capabilities via Agent Cards.
- Negotiate interaction modalities (text, files, structured data).
- Securely collaborate on long-running tasks.
- Operate without exposing internal state, memory, or tools (opaque execution).

### Guiding Principles

- **Simple:** Reuses existing standards — HTTP(S), JSON-RPC 2.0, Server-Sent Events, gRPC.
- **Enterprise Ready:** Addresses authentication, authorization, security, privacy, tracing, and monitoring.
- **Async First:** Designed for long-running tasks and human-in-the-loop interactions.
- **Modality Agnostic:** Supports text, audio/video (via file references), structured data/forms, and embedded UI components.
- **Opaque Execution:** Agents collaborate based on declared capabilities without sharing internal thoughts, plans, or tool implementations.

### Three-Layer Architecture

A2A is organized into three layers:

1. **Canonical Data Model** — Core data structures (Task, Message, AgentCard, Part, Artifact) defined as Protocol Buffers.
2. **Abstract Operations** — Fundamental capabilities independent of transport (SendMessage, GetTask, CancelTask, etc.).
3. **Protocol Bindings** — Concrete mappings to JSON-RPC 2.0, gRPC, and HTTP+JSON/REST.

### SDKs

Official SDKs are available in Python (`a2a-sdk`), Go, JavaScript/TypeScript (`@a2a-js/sdk`), Java (Maven), and .NET (NuGet package `A2A`).

## When to Use

- Building multi-agent systems where independent agents need to collaborate as peers.
- Creating agent-to-agent communication layers (as opposed to agent-to-tool, which is MCP's domain).
- Implementing agent discovery via Agent Cards at well-known URIs.
- Managing long-running tasks with streaming (SSE) or push notification (webhook) updates.
- Building enterprise-grade agent systems requiring authentication, authorization, and observability.
- Integrating agents across different frameworks (Google ADK, LangGraph, BeeAI, etc.).

## Core Concepts

### Key Actors

- **User:** A human operator or automated service initiating a request.
- **A2A Client (Client Agent):** An application or agent that initiates requests to an A2A Server.
- **A2A Server (Remote Agent):** An agent exposing an A2A-compliant endpoint. Operates as an opaque (black-box) system — internal workings, memory, and tools are not exposed.

### Fundamental Objects

- **Agent Card:** A JSON metadata document describing an agent's identity, capabilities, skills, service endpoints, and authentication requirements. Published at `/.well-known/agent-card.json`.
- **Task:** The fundamental unit of work. Stateful, identified by a unique server-generated ID, progresses through a defined lifecycle (working → completed/failed/canceled/rejected/input-required/auth-required).
- **Message:** A communication turn between client and agent with a role (`ROLE_USER` or `ROLE_AGENT`) containing one or more Parts.
- **Part:** The smallest content unit within a Message or Artifact. Can contain text, file references (URL or inline bytes), or structured JSON data.
- **Artifact:** Tangible output generated by an agent during task processing (documents, images, structured data). Composed of Parts.

### Interaction Mechanisms

- **Request/Response (Polling):** Client sends a request and polls for updates via GetTask.
- **Streaming (SSE):** Real-time incremental updates over persistent HTTP connections. Requires `capabilities.streaming: true`.
- **Push Notifications (Webhooks):** Server-initiated HTTP POST to client-provided webhook URLs. Requires `capabilities.pushNotifications: true`.

### Task Lifecycle

Tasks progress through states:

- **Working:** Agent is actively processing.
- **Completed:** Task finished successfully.
- **Failed:** Task encountered an unrecoverable error.
- **Canceled:** Task was canceled by the client.
- **Rejected:** Agent rejected the task.
- **Input Required:** Agent needs additional input from the client (multi-turn).
- **Auth Required:** Agent needs authorization credentials to proceed.

Terminal states (completed, failed, canceled, rejected) are immutable — tasks cannot restart.

### Context and Multi-Turn

A `contextId` logically groups related Tasks and Messages, providing conversational continuity. Clients can:

- Continue a specific task by including both `taskId` and `contextId`.
- Start a new task within an existing conversation using only `contextId`.
- Reference previous tasks via `referenceTaskIds` for follow-ups and refinements.

## Protocol Bindings

A2A supports three standard protocol bindings:

### JSON-RPC 2.0

HTTP(S) with JSON-RPC 2.0 method calls and Server-Sent Events for streaming. Content-Type: `application/json`. Methods use PascalCase (e.g., `SendMessage`, `GetTask`).

### gRPC

Protocol Buffers over HTTP/2 with TLS. Uses the normative `a2a.proto` definition. Service parameters transmitted via gRPC metadata (headers).

### HTTP+JSON/REST

RESTful endpoints using standard HTTP verbs. Content-Type: `application/a2a+json`. Streaming via SSE.

### Method Mapping

- Send message → `SendMessage` / `POST /message:send`
- Send streaming message → `SendStreamingMessage` / `POST /message:stream`
- Get task → `GetTask` / `GET /tasks/{id}`
- List tasks → `ListTasks` / `GET /tasks`
- Cancel task → `CancelTask` / `POST /tasks/{id}:cancel`
- Subscribe to task → `SubscribeToTask` / `POST /tasks/{id}:subscribe`
- Create push notification config → `CreateTaskPushNotificationConfig` / `POST /tasks/{id}/pushNotificationConfigs`
- Get extended Agent Card → `GetExtendedAgentCard` / `GET /extendedAgentCard`

## Usage Examples

### Sending a Message (HTTP+JSON)

```http
POST /message:send HTTP/1.1
Host: agent.example.com
Content-Type: application/a2a+json
Authorization: Bearer token
A2A-Version: 1.0

{
  "message": {
    "role": "ROLE_USER",
    "parts": [{"text": "What is the weather today?"}],
    "messageId": "msg-uuid"
  }
}
```

**Response:**

```json
{
  "task": {
    "id": "task-uuid",
    "contextId": "context-uuid",
    "status": {"state": "TASK_STATE_COMPLETED"},
    "artifacts": [{
      "artifactId": "artifact-uuid",
      "name": "Weather Report",
      "parts": [{"text": "Today will be sunny with a high of 75°F"}]
    }]
  }
}
```

### Discovering an Agent Card

```http
GET /.well-known/agent-card.json HTTP/1.1
Host: agent.example.com
```

Returns a JSON document with the agent's name, description, supported interfaces, capabilities, skills, and security requirements.
