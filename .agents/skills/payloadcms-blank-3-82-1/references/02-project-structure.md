# Project Structure

Detailed explanation of the Payload CMS blank template directory structure, file purposes, and organization patterns.


## Source Directory (src/)

### Main Configuration Files

```
src/
├── payload.config.ts        # Main Payload configuration
└── payload-types.ts         # Auto-generated TypeScript types
```

**payload.config.ts:**
- Central configuration for all Payload settings
- Imports and registers collections, globals, plugins
- Configures database adapter, editor, admin panel
- Defines secret key, TypeScript options, sharp instance
- **Never import this directly in components** (use `@payload-config` alias)

**payload-types.ts:**
- Auto-generated by `pnpm generate:types`
- Contains TypeScript types for all collections and globals
- Includes field types, relationship types, hook contexts
- **Never edit manually** - regenerate after schema changes
- Import types from here: `import type { User, Post } from '@/payload-types'`

### Collections Directory

```
src/collections/
├── Users.ts                 # Auth-enabled user collection
└── Media.ts                 # Upload-enabled media collection
```

**Collections:**
- Each file exports a `CollectionConfig` object
- Slug matches filename (e.g., `Users.ts` → slug: 'users')
- Define fields, access control, hooks, admin options
- Import and register in `payload.config.ts`

See [Collection Configuration](03-collection-configuration.md) for detailed collection patterns.

### App Directory (Next.js App Router)

```
src/app/
├── (payload)/               # Payload admin panel route group
│   ├── admin/
│   │   └── [[...segments]]/ # Catch-all route for admin pages
│   │       ├── page.tsx     # Admin panel root component
│   │       └── not-found.tsx # 404 handler for admin
│   ├── api/
│   │   ├── [...slug]/route.ts    # REST API endpoint
│   │   └── graphql/
│   │       ├── route.ts           # GraphQL API endpoint
│   │       └── playground/route.ts # GraphQL Playground (dev only)
│   └── layout.tsx         # Admin panel layout
└── (frontend)/             # Public frontend route group
    ├── page.tsx           # Home page
    ├── layout.tsx         # Frontend root layout
    └── styles.css         # Global frontend styles
```

### Route Groups

**Route groups** (directories in parentheses) organize routes without affecting URL structure:

- `(payload)/admin/` → URL: `/admin/`
- `(frontend)/` → URL: `/` (root)
- `(frontend)/page.tsx` → URL: `/`

#### Admin Panel Routes (`(payload)/admin/`)

**[[...segments]]/page.tsx:**
```typescript
/* THIS FILE WAS GENERATED AUTOMATICALLY BY PAYLOAD. */
/* DO NOT MODIFY IT BECAUSE IT COULD BE REWRITTEN AT ANY TIME. */
import type { Metadata } from 'next'

import config from '@payload-config'
import { RootPage, generatePageMetadata } from '@payloadcms/next/views'
import { importMap } from '../importMap'

type Args = {
  params: Promise<{
    segments: string[]
  }>
  searchParams: Promise<{
    [key: string]: string | string[]
  }>
}

export const generateMetadata = ({ params, searchParams }: Args): Promise<Metadata> =>
  generatePageMetadata({ config, params, searchParams })

const Page = ({ params, searchParams }: Args) =>
  RootPage({ config, params, searchParams, importMap })

export default Page
```

**Purpose:**
- Renders Payload admin panel at all `/admin/*` routes
- `RootPage` component handles routing to collections, globals, settings
- `[[...segments]]` captures all nested paths (e.g., `/admin/collections/users/123`)
- Auto-generated - do not modify manually

**importMap.js:**
```javascript
// app/(payload)/admin/importMap.js
// Auto-generated component import map
export const importMap = {
  // Maps component paths to actual imports
}
```

Regenerate with: `pnpm generate:importmap`

#### API Routes (`(payload)/api/`)

**REST API ([...slug]/route.ts):**
```typescript
/* THIS FILE WAS GENERATED AUTOMATICALLY BY PAYLOAD. */
import config from '@payload-config'
import '@payloadcms/next/css'
import {
  REST_DELETE,
  REST_GET,
  REST_OPTIONS,
  REST_PATCH,
  REST_POST,
  REST_PUT,
} from '@payloadcms/next/routes'

export const GET = REST_GET(config)
export const POST = REST_POST(config)
export const DELETE = REST_DELETE(config)
export const PATCH = REST_PATCH(config)
export const PUT = REST_PUT(config)
export const OPTIONS = REST_OPTIONS(config)
```

**Purpose:**
- Handles all REST API requests at `/api/*`
- Routes to appropriate collection/operation based on URL
- Examples:
  - `GET /api/users` → List users
  - `POST /api/users` → Create user
  - `GET /api/users/123` → Get user by ID
  - `PATCH /api/users/123` → Update user
  - `DELETE /api/users/123` → Delete user

**GraphQL API (graphql/route.ts):**
```typescript
import config from '@payload-config'
import { GRAPHQL_POST, REST_OPTIONS } from '@payloadcms/next/routes'

export const POST = GRAPHQL_POST(config)
export const OPTIONS = REST_OPTIONS(config)
```

**Purpose:**
- Handles GraphQL queries/mutations at `/api/graphql`
- Only accepts POST requests (GraphQL standard)
- Auto-generated schema from collections/globals

**GraphQL Playground (graphql-playground/route.ts):**
- Development-only interactive GraphQL IDE
- Accessible at `/api/graphql-playground` in development
- Disabled in production by default

#### Frontend Routes (`(frontend)/`)

**page.tsx (Home Page):**
```typescript
import { headers as getHeaders } from 'next/headers.js'
import Image from 'next/image'
import { getPayload } from 'payload'
import React from 'react'

import config from '@/payload.config'
import './styles.css'

export default async function HomePage() {
  const headers = await getHeaders()
  const payloadConfig = await config
  const payload = await getPayload({ config: payloadConfig })
  const { user } = await payload.auth({ headers })

  return (
    <div className="home">
      <div className="content">
        {!user && <h1>Welcome to your new project.</h1>}
        {user && <h1>Welcome back, {user.email}</h1>}
        <div className="links">
          <a href={payloadConfig.routes.admin}>Go to admin panel</a>
          <a href="https://payloadcms.com/docs">Documentation</a>
        </div>
      </div>
    </div>
  )
}
```

**Purpose:**
- Server component (async function)
- Uses Local API to check authentication
- Conditional rendering based on user login
- Starting point for custom frontend pages

**layout.tsx (Root Layout):**
```typescript
import React from 'react'
import './styles.css'

export const metadata = {
  description: 'A blank template using Payload in a Next.js app.',
  title: 'Payload Blank Template',
}

export default async function RootLayout(props: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <main>{children}</main>
      </body>
    </html>
  )
}
```

**Purpose:**
- Wraps all frontend pages
- Defines HTML structure and metadata
- Imports global styles
- Persistent across route changes

**styles.css:**
- Global CSS for frontend
- Custom properties (CSS variables)
- Responsive typography
- Utility classes

See [Custom Routes and Endpoints](05-custom-routes-endpoints.md) for creating custom API routes.

### E2E Tests (Playwright)

**admin.e2e.spec.ts:**
- Tests admin panel functionality
- Verifies navigation, forms, list views
- Requires running dev server
- Uses real browser (Chromium)

**frontend.e2e.spec.ts:**
- Tests public frontend pages
- Verifies rendering, interactions
- Cross-browser testing capability

### Integration Tests (Vitest)

**api.int.spec.ts:**
- Tests Local API operations
- Faster than E2E (no browser)
- Uses jsdom environment
- Tests business logic, hooks, access control

### Test Helpers

**login.ts:**
```typescript
export async function login({ page, user }: LoginOptions): Promise<void> {
  await page.goto(`${serverURL}/admin/login`)
  await page.fill('#field-email', user.email)
  await page.fill('#field-password', user.password)
  await page.click('button[type="submit"]')
  await page.waitForURL(`${serverURL}/admin`)
}
```

**seedUser.ts:**
```typescript
export const testUser = {
  email: 'dev@payloadcms.com',
  password: 'test',
}

export async function seedTestUser(): Promise<void> {
  const payload = await getPayload({ config })
  await payload.create({ collection: 'users', data: testUser })
}

export async function cleanupTestUser(): Promise<void> {
  const payload = await getPayload({ config })
  await payload.delete({
    collection: 'users',
    where: { email: { equals: testUser.email } },
  })
}
```

## Type Generation

### When to Run

Generate types **after any schema change**:
- Adding/removing collections
- Modifying collection fields
- Adding/removing globals
- Changing field types or options
- Updating access control functions

### How to Run

```bash
# Generate TypeScript types
pnpm generate:types
```

This creates/updates `src/payload-types.ts` with:

```typescript
// Auto-generated types
export type User = {
  id: string
  email: EmailField
  resetPasswordToken?: ResetPasswordToken
  resetPasswordExpiration?: Date
  salt?: string
  hash?: string
  refreshToken?: string
  createdAt: string
  updatedAt: string
}

export type Media = {
  id: string
  alt: string
  url?: string
  filename?: string
  mimeType?: string
  filesize?: number
  width?: number
  height?: number
  focalX?: number
  focalY?: number
  createdAt: string
  updatedAt: string
}

// Relationship types, hook contexts, etc.
```

### Using Generated Types

**Import types:**
```typescript
import type { User, Post } from '@/payload-types'

function displayUser(user: User) {
  console.log(user.email) // Type-safe
}
```

**Use in hooks:**
```typescript
import type { BeforeChangeHook } from 'payload'

const beforeChange: BeforeChangeHook<User> = async ({ data }) => {
  // data is typed as User
  return {
    ...data,
    email: data.email.toLowerCase(),
  }
}
```

**Type guards for field access:**
```typescript
import type { Post } from '@/payload-types'

if ('author' in post && post.author) {
  // post.author is guaranteed to exist
  console.log(post.author.email)
}
```

### Troubleshooting Type Generation

**"Cannot find module '@payload-config'"**:
- Delete `.next` directory: `rm -rf .next`
- Regenerate types: `pnpm generate:types`
- Restart development server

**TypeScript errors in payload-types.ts**:
- Check for syntax errors in collection configs
- Verify all imports resolve correctly
- Ensure Payload and Next.js versions are compatible

**Missing types after generation**:
- Verify collection is registered in `payload.config.ts`
- Check collection slug matches import name
- Restart TypeScript server (VS Code: Cmd+Shift>P > TS: Restart)

## Path Aliases

### Configuration (tsconfig.json)

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"],
      "@payload-config": ["./src/payload.config.ts"]
    }
  }
}
```

### Usage Examples

**@/ alias (src directory):**
```typescript
// Instead of:
import { Users } from '../../collections/Users'
import config from '../../payload.config'

// Use:
import { Users } from '@/collections/Users'
import config from '@payload-config'
```

**Importing from different directories:**
```typescript
// Collections
import { Media } from '@/collections/Media'

// Components
import CustomField from '@/components/CustomField'

// Hooks
import { beforeChangeSlug } from '@/hooks/beforeChangeSlug'

// Access control
import { adminOnly } from '@/access/adminOnly'

// Utilities
import { slugify } from '@/utils/slugify'
```

### Path Resolution in Tests

Vitest and Playwright need configuration to resolve paths:

**vitest.config.mts:**
```typescript
import tsconfigPaths from 'vite-tsconfig-paths'

export default defineConfig({
  plugins: [tsconfigPaths()], // Enables path resolution
})
```

**playwright.config.ts:**
- Uses `tsx` for TypeScript execution
- Automatically resolves paths via tsconfig.json

### When to Extract to Separate Files

**Extract access control when:**
- Used across multiple collections
- Logic is complex or multi-step
- Needs to be tested independently

**Extract hooks when:**
- Reused across multiple fields/collections
- Contains significant business logic
- Benefits from unit testing

**Extract components when:**
- Customizing admin panel UI
- Creating reusable field components
- Building custom views (dashboard, list headers)

See [Collection Configuration](03-collection-configuration.md) and [Security Patterns](04-security-patterns.md) for detailed examples.
